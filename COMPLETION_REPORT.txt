✅ TASK 2 — RETURN FLOW
───────────────────────────────────────────────────────────────────────────────
Status: COMPLETE & VERIFIED

Return Process Flow:
  1. Shop creates return request → Status: PENDING
     (No inventory update at this stage)
  2. Admin reviews and approves/rejects → Status: APPROVED/REJECTED
  3. If APPROVED → Inventory increases (return_in transaction)
  4. If REJECTED → Remains in database for records

Backend Implementation:
  ✅ POST /api/return - Create return request
  ✅ GET /api/return - List returns (shop-specific via role)
  ✅ GET /api/return/:id - Get return details
  ✅ PUT /api/return/:id/status - Update return status (admin only)
  ✅ DELETE /api/return/:id - Delete return (only pending)
  ✅ GET /api/return/pending/count - Count pending returns

Return Model (Return.js):
  - returnNo: Auto-generated unique ID
  - saleId: Link to original sale
  - shopId: Which shop returning
  - items: Array of returned products with reasons
  - reason: damaged/expired/excess
  - status: pending/approved/rejected
  - approvedDate, approvedBy, rejectionReason

Inventory Service Integration:
  When return is APPROVED:
  → Call inventoryService.updateInventory()
    * shopId, productId, +quantity
    * transactionType: 'return_in'
    * Creates StockLedger entry

Frontend Implementation:
  ✅ Shop Returns page (/shop/returns)
     - Form to create return requests
     - Select product from inventory
     - Enter quantity & reason
     - Submit return
  ✅ Admin Returns page (/admin/returns)
     - View all pending returns
     - Approve/reject button
     - Add rejection reason if needed
     - View complete return history

API Calls in adminApi:
  ✅ getReturns(shopId) - List returns
  ✅ getPendingReturns() - Count pending
  ✅ getReturnById(id) - Get details
  ✅ updateReturnStatus(id, data) - Approve/reject

✅ TASK 3 — INVENTORY VALIDATION
───────────────────────────────────────────────────────────────────────────────
Status: COMPLETE & VERIFIED

Overselling Prevention:
  File: server/src/controllers/salesController.js (lines 41-52)

Validation Logic:
  1. Before creating sale, loop through all items
  2. For each item, check Inventory record
  3. Compare requestedQty against availableQty
  4. If requestedQty > availableQty:
     → Return 400 error with descriptive message
     → Sale NOT created
     → Inventory NOT updated

Error Message Format:
  "Insufficient stock for Full Cream Milk. Available: 50, Requested: 100"

Implementation Details:
  const availableQty = inventory?.quantity || 0;
  if (availableQty < item.quantity) {
    return responseHelper.error(res, `Insufficient stock for ${item.productName}...`, 400);
  }

Test Verification:
  ✅ Attempted oversale: BLOCKED
  ✅ Valid sale (qty < available): ACCEPTED
  ✅ Payment not processed if validation fails
  ✅ Inventory remains unchanged on failed validation
